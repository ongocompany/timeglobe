# 개발 기본 지침서: TimeGlobe 기획 및 개발 컨텍스트

## 1. 프로젝트 개요
* **프로젝트명:** TimeGlobe (가칭)
* **목적:** 지질학적 시대부터 인류 역사까지 지구의 공간적·시간적 변화를 3D 지구본과 타임라인으로 탐험하는 인터랙티브 에듀테크 서비스.
* **타겟 사용자:** 역사/과학 관심층, 학생, 교육 종사자 및 자녀를 둔 학부모.
* **개발 방식:** 1인 개발, AI 코딩 도구(바이브 코딩) 적극 활용.
* **플랫폼:** 반응형 웹(Web) 우선 개발 후, 향후 모바일 앱(App)으로 패키징 확장 고려.

## 2. 주요 기술 스택
* **프론트엔드:** Next.js (확정)
* **3D 렌더링:** `Globe.gl` (우주 배경, 3D 지구본, 마커, 텍스처 맵 및 GeoJSON 국경선 표현)
* **백엔드/데이터베이스:** Supabase (동적 데이터 관리, 사용자 인증 및 결제 정보 처리)
* **배포:** Vercel (웹) / Capacitor 또는 PWA (모바일 앱)

## 3. 핵심 차별점 및 비즈니스 모델
* **통합된 경험:** 기존 서비스들이 분리해 놓은 '지질 시대(대륙 이동)'와 '인류 역사(문명/국경)'를 3D 지구본 하나로 매끄럽게 연결.
* **선택과 집중 (큐레이션):** 방대한 데이터를 단순 나열하지 않고, 핵심 사건 위주의 직관적인 마커 제공. 서구권 중심에서 벗어나 삼국시대, 當時(당시)의 이백/두보 등 동양사 데이터의 디테일 강화.
* **수익 모델 (Freemium):** MVP 제한 노출(예: 서기 0~1500년 구간 무료) 후, 지질 시대 및 근현대사 풀 콘텐츠는 '일회성 결제(Time-Travel Pass)'로 잠금 해제.
* **게이미피케이션 (학부모 타겟 결제 유도):** 
    * 타임 트래블러 패스포트 (도감 스탬프 수집).
    * 인터랙티브 역사 퀴즈 (위치 찾기 등) 및 부모님용 학습 진척도 확인 기능.
    * 유저 레벨링 시스템 (`travel_tokens` 기반 '초보 타임트래블러' -> '타임로드' 등급 부여)

## 4. UI 및 데이터 시각화 전략
* **마커 카테고리:** 정치/전쟁(빨강), 인물/문화(노랑), 과학/발명(파랑), 건축/유물(초록), 자연재해/지질(회색) 등으로 시각적 구분.
* **불명확한 지역:** 중남미 문명 등 국경이 모호한 곳은 Heatmap 스타일의 원형 마커나 안개(Fog) 효과로 뉘앙스만 표현.
* **몰입감 강화:** 시대/지역별 동적 배경음악 적용 및 화산 폭발 등 주요 이벤트 파티클 이펙트.
* **다국어 지원:** JSONB를 활용한 핵심 텍스트 데이터 다국어 확장성 확보.

## 5. 팝업 모달(Modal) 필수 구성 요소 (데이터베이스 설계 기준)
해당 마커를 클릭했을 때 나타나는 상세 정보 창.
* **헤더:** 이벤트 제목, 발생 연도/시대, 카테고리 태그.
* **본문:** 시각적 몰입을 위한 대표 이미지 1장, 핵심 요약 설명(3~4줄), 현재 국가/지역명 텍스트.
* **액션(하단):** 나무위키 등 외부 링크 버튼, 도감(패스포트) 스탬프 획득 알림, 미니 퀴즈 시작 버튼, 관련 시대 이벤트 추천.

## 6. 기술 검토 및 추가 제안 (2026-02-24)

### 6-1. [cl] 프론트엔드 프레임워크 결정 (Next.js 채택)
현재 "React (또는 Next.js)"로 미정 상태. **Next.js 확정을 권장**하는 이유:
* **SEO**: 에듀테크 서비스 특성상 "삼국시대 역사", "로마제국 지도" 등 검색 유입이 중요. SPA만으로는 SEO가 약함.
* **Vercel 배포 궁합**: Next.js + Vercel은 같은 회사 제품으로 배포가 압도적으로 편리.
* **SSG 활용**: Events, Eras 등 정적 데이터는 빌드 시 미리 생성(Static Site Generation)해두면 초기 로딩 속도 향상.
**-> [jn/gm/cl 합의 완료] Next.js 도입 확정.**

### 6-2. [gm] Globe.gl 성능 리스크 및 오픈소스 대안 분석
Globe.gl은 WebGL(Three.js) 기반이라 수백 개의 마커와 무거운 GeoJSON 폴리곤(국경선) 동시 렌더링 시 모바일 버벅임이나 메모리 이슈가 생길 위험이 높습니다. 이를 대체할 만한 고성능 지도 렌더링 오픈소스 라이브러리는 다음과 같습니다.

1. **deck.gl (가장 추천)**
   * **특징**: Uber에서 개발한 WebGL2 기반 고성능 대용량 데이터 시각화 프레임워크.
   * **장점**: React 환경과 100% 찰떡궁합. 자체적으로 `GeoJsonLayer`를 완벽 지원하며, 대규모 폴리곤이나 마커를 렌더링할 때 Globe.gl보다 성능이 압도적으로 우수함. 최근부터 3D Globe 뷰(GlobeView)도 지원.
   * **결론**: 프론트엔드를 통째로 직접 구현하지 않아도 되면서 성능 리스크를 덜어줄 가장 강력한 대안.

2. **CesiumJS**
   * **특징**: 고정밀 3D 지구본/지도 렌더링의 끝판왕 격인 오픈소스.
   * **장점**: 막강한 리얼리즘, 정교한 3D 데이터 스트리밍 특화.
   * **단점**: 무겁고, React 연동(`Resium`)이 deck.gl만큼 네이티브하지 않으며, 무거운 GeoJSON을 그대로 로드하기보단 전용 포맷(3D Tiles)으로 변환해 줘야 빠름.

**-> [gm의 제안 및 DB 스키마 수정안]**
* 라이브러리 직접 렌더링 구현은 Three.js 바닥부터 다 짜야 해서 1인 개발 시간 낭비성 리스크가 큽니다.
* 현 단계에서는 **`deck.gl`의 GlobeView + GeoJsonLayer 조합**을 1순위 대안 기술로 올려두고 리서치/PoC를 진행하는 것이 좋겠습니다.
* **DB 스키마 변경 사항**: 렌더링 부하를 줄이기 위해 어떤 라이브러리를 쓰든 대규모 GeoJSON 텍스트를 Supabase DB(`jsonb`)에서 매번 꺼내오는 구조는 위험합니다. 따라서 데이터베이스의 `Borders` 테이블 설계에서 `geojson_data(jsonb)` 필드를 삭제하고, **Cloudflare R2나 Vercel 외부 CDN에 `.geojson` 정적 파일을 올려둔 뒤 그 주소를 가리키는 `geojson_url(text)` 컬럼으로 변경하도록 제안합니다.**
