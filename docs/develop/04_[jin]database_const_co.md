3) 데이터 수집의 핵심 목표 (품질 기준)

이 서비스는 학술 DB 구축이 아니라 가볍고 빠르게 탐색/비교/학습하는 역사 콘텐츠다.
따라서 수집 파이프라인의 목표는 다음과 같다.

우선순위

구조화 가능성

다국어 대응 가능성

운영 안정성

확장성

정밀도(학술급 완전성보다 UX 적합성)

즉, “완벽한 역사 데이터”보다 지속적으로 수집·보정 가능한 데이터셋이 중요함.

4) 수집 전략 상세 (Codex가 따라야 할 방향)
4-1) 1차 소스: Wikidata (핵심)

Wikidata에서 아래 성격의 데이터를 수집한다.

엔티티 식별자(QID)

엔티티 타입(사건/인물/장소/시대 등)

시간 정보(시작/종료/출생/사망 등)

좌표 정보(가능한 경우)

국가/상위 지역 연결(가능한 경우)

다국어 label / description / aliases

Wikipedia sitelink 정보(언어별 페이지 연결용)

원칙

구조화 필드 우선

텍스트 본문 해석 최소화

타입 분류/정규화는 파이프라인에서 수행

4-2) 2차 소스: Wikipedia (보조)

Wikipedia는 다음 용도로만 사용한다.

카드용 짧은 요약 텍스트

대표 이미지(썸네일)

링크 연결

원칙

본문 전체 저장 금지(또는 최소화)

필요 시 캐시 가능하되, “표현 보조 데이터”로 취급

구조의 기준은 항상 Wikidata

4-3) 수집 범위(MVP)

초기에는 전 범위를 한 번에 과도하게 수집하지 말고, 아래처럼 시작한다.

엔티티 우선순위

Event (사건)

Person (인물)

Place (장소)

(선택) Era / Dynasty / Polity

언어 우선순위

1차: ko, en

확장 준비: ja, zh

실행 단위

기간별 분할 수집 (예: 세기/연대 단위)

타입별 분리 수집

필요 시 지역 기준 분할 수집

5) 실행 방식 (중요)

리포트 출력

6) 운영 관점 요구사항 (필수)

Codex는 아래 운영 요소를 반드시 포함해줘.

6-1) 체크포인트/재시작 가능

실행 중단 후 다시 돌려도 이어서 진행 가능해야 함

task 단위 상태 저장 (pending / success / failed)

6-2) 재시도 로직

WDQS 응답 실패 / timeout / 일시 오류 시 재시도

exponential backoff 권장

6-3) 과부하 방지

요청 간 딜레이 / 동시성 제한

큰 기간 범위를 자동 분할할 수 있으면 좋음

6-4) 로깅

최소한 다음 로그 필요:

어떤 task 실행 중인지

몇 건 수집했는지
